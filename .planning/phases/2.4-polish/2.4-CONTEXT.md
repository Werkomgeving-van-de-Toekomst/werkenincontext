# Phase 2.4: Polish - Context

**Gathered:** 2026-03-08
**Status:** Ready for planning

## Phase Boundary

Complete production-ready UX with URL state sharing (filter and view state persist in URL), smooth animations for view transitions, and consistent design system styling across all 3D controls.

## Implementation Decisions

### URL State Format
- **Readable query params** (e.g., `?year=1950-2020&view=3d`)
- Clean, shareable URLs that users can understand and edit
- State changes update browser history (user can navigate back)
- Include all filter parameters: year range, height range, floor count, view mode, density heatmap toggle

### Transition Timing
- **500ms camera** (MapLibre `easeTo`) — matches existing implementation
- **300ms UI** (CSS transitions) for staggered effect
- UI elements fade slightly faster than camera moves for polished feel
- All animations use easing (not linear) for natural motion

### Styling Approach
- **Use existing CSS classes consistently** — leverage `--flevoland-blauw`, `--spacing-*`, `--radius-*` variables
- Replace inline styles with existing classes: `.btn-primary`, `.btn-secondary`, `.panel`, `.panel-header`
- Create new CSS classes only where existing ones don't fit (e.g., `.filter-panel-3d`, `.view-toggle-container`)
- Remove hardcoded colors and spacing from component Rust code

### Browser Support
- **Modern browsers**: Chrome, Firefox, Safari
- Test URL encoding/decoding across browsers
- Verify CSS transitions work smoothly
- No IE11 or legacy browser support needed

### Claude's Discretion
- Exact URL parameter names (e.g., `year` vs `bouwjaar`, `view` vs `weergave`)
- Whether to use `history.replaceState()` vs `history.pushState()` for state updates
- Loading state handling during URL restoration
- Error handling for malformed URL parameters

## Specific Ideas

- "I like how the existing view toggle uses localStorage — URL should complement this, not replace it"
- URL state should be the source of truth when sharing, localStorage for return visits
- Transitions should feel like the existing easeTo (500ms) — don't reinvent, just extend
