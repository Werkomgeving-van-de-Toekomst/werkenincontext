# Phase 2.4: Polish - Research

**Researched:** 2026-03-08
**Domain:** Dioxus 0.7 WASM URL State Persistence + CSS Transitions + Design System Styling
**Confidence:** MEDIUM

## Summary

Phase 2.4 focuses on production-ready UX polish: URL state sharing for filters/views, smooth view transitions, and consistent design system styling. The key technical challenges are: (1) serializing Rust state structs to URL query parameters in Dioxus WASM, (2) adding CSS transitions to MapLibre view changes, and (3) extending the existing Provincie Flevoland design system across all 3D building controls.

**Primary recommendation:** Use serde + URL encoding for state serialization to query params, MapLibre's `easeTo()` for smooth camera transitions, and leverage existing CSS variables from `style.css` for consistent styling.

### Pattern 1: URL State Serialization via JS Interop

**What:** Serialize Rust state to URL query params using `pushState()`

**When to use:** Any shareable application state (filters, view modes, toggles)

**Why this pattern:** Dioxus 0.7 router's query parameter support is not well-documented; the codebase already uses `document::eval()` for JS interop (see `view_toggle.rs`, `filter_panel_3d.rs`).

**Example:**

```rust
// Source: Based on existing view_toggle.rs localStorage pattern (HIGH confidence)
use serde_json;
use urlencoding;

#[derive(Serialize, Deserialize)]
pub struct UrlState {
    pub view_mode: ViewMode,
    pub filter: BuildingFilter,
    pub heatmap: DensityHeatmap,
}

pub fn build_update_url_script(state: &UrlState) -> String {
    let json = serde_json::to_string(state).expect("Serialization failed");
    let encoded = urlencoding::encode(&json);

    format!(
        r#"
        (function() {{
            const state = {};
            const encoded = '{}';
            const url = new URL(window.location);
            url.searchParams.set('state', encoded);
            window.history.replaceState({{state}}, '', url);
            console.log('URL updated:', url.toString());
        }})();
        "#,
        json, encoded
    )
}
```

### Pattern 2: URL State Restoration on Mount

**What:** Parse URL query params on page load and restore state

**When to use:** Component initialization in `data_verkenner.rs` or individual controls

**Example:**

```rust
// Source: Based on view_toggle.rs build_get_initial_view_script() pattern (HIGH confidence)
pub fn build_restore_state_script() -> String {
    r#"
    (function() {
        try {
            const params = new URLSearchParams(window.location.search);
            const encoded = params.get('state');
            if (encoded) {
                const decoded = decodeURIComponent(encoded);
                const state = JSON.parse(decoded);
                window['initialUrlState'] = state;
                console.log('Restored state from URL:', state);
            } else {
                window['initialUrlState'] = null;
            }
        } catch (e) {
            console.error('Error restoring URL state:', e);
            window['initialUrlState'] = null;
        }
    })();
    "#.to_string()
}
```

### Pattern 3: Smooth View Transitions

**What:** Animate MapLibre camera and UI elements during 2D/3D toggle

**When to use:** Any view state change (2D/3D toggle, filter changes)

**Example:**

```javascript
// Source: Existing view_toggle.rs already uses easeTo() (HIGH confidence)
function setViewModeWithTransition(mode) {
    const map = window['map_map'];
    if (!map) return;

    const pitch = mode === '3d' ? 60 : 0;
    const duration = 500; // 500ms smooth transition

    // Smooth camera transition
    map.easeTo({
        pitch: pitch,
        duration: duration,
        easing: (t) => t < 0.5 ? 2 * t * t : -1 + (4 - 2 * t) * t // ease-in-out
    });

    // Set building height (instant, no animation)
    const height = mode === '3d'
        ? ["coalesce", ["get", "height"], 10]
        : 0;
    map.setPaintProperty('building-3d', 'fill-extrusion-height', height);
}
```

### Anti-Patterns to Avoid

- **Inline styles for colors:** Don't use `#0066CC` directly; use `var(--flevoland-blauw)` for consistency
- **Hardcoded spacing:** Don't use `padding: 8px`; use `padding: var(--spacing-sm)`
- **Missing transition:** Don't change state without animation; use `transition` property
- **URL bloat:** Don't encode entire state on every change; use `replaceState()` for existing state, `pushState()` only for navigation
- **Blocking UI:** Don't let URL serialization block render; run in background effect

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| URL parsing | Manual string splitting | `URLSearchParams` + `urlencoding` crate | Handles encoding, special chars, edge cases |
| JSON serialization | Manual format | `serde_json` | Already implemented for all state structs |
| Camera animation | Custom requestAnimationFrame | MapLibre `easeTo()` | GPU-accelerated, handles easing functions |
| Color variables | Hardcoded hex values | CSS variables from `style.css` | Consistent with design system, themeable |

## Common Pitfalls

### Pitfall 1: URL State Desync

**What goes wrong:** URL shows one state, UI shows another (e.g., after browser back button)

**Why it happens:** URL updates via `pushState()` but Rust signals don't re-synchronize

**How to avoid:**
- Use `replaceState()` for state updates (doesn't add history entries)
- Listen to `popstate` event to restore state on back/forward
- Initialize state from URL on component mount

**Warning signs:** Back button changes URL but not UI, URL grows indefinitely with every state change

### Pitfall 2: JS → Rust State Sync

**What goes wrong:** URL state restored in JS but Rust signals not updated

**Why it happens:** Dioxus WASM interop is one-way (Rust → JS), no built-in JS → Rust channel

**How to avoid:**
- Parse URL params in Rust during component initialization
- Use `use_effect` to read from `window.location.search` via JS
- Consider using `web-sys` `window().location().search()` directly in Rust if available

### Pitfall 3: Transition Jank

**What goes wrong:** Map animates smoothly but UI controls jump

**Why it happens:** CSS transitions not applied to UI elements, or transition duration mismatch

**How to avoid:**
- Apply `transition` property to all animated elements
- Match CSS transition duration (e.g., `0.3s`) to JS animation duration (e.g., `300ms`)
- Use `transform` and `opacity` (GPU-accelerated) instead of `left`, `top`, `width`, `height`

**Warning signs:** Elements snap instead of slide, frame rate drops during transition

### Pitfall 4: Design System Drift

**What goes wrong:** New controls use different colors/spacing than existing UI

**Why it happens:** Copying styles from elsewhere instead of using CSS variables

**How to avoid:**
- Audit all inline styles in `filter_panel_3d.rs`, `view_toggle.rs`, `density_heatmap.rs`
- Create shared CSS class in `style.css` or new `controls.css`
- Use CSS variables for all colors, spacing, borders

## Code Examples

### URL State Management Module

```rust
// Source: New module for phase 2.4 (HIGH confidence based on existing patterns)
//! URL state serialization and restoration

use serde::{Deserialize, Serialize};
use crate::state::{BuildingFilter, ViewMode, DensityHeatmap};

/// Complete application state for URL sharing
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct UrlState {
    #[serde(default)]
    pub view_mode: ViewMode,
    #[serde(default)]
    pub filter: BuildingFilter,
    #[serde(default)]
    pub heatmap: DensityHeatmap,
}

impl UrlState {
    /// Serialize to URL-safe base64 encoded JSON
    pub fn to_url_param(&self) -> String {
        let json = serde_json::to_string(self)
            .expect("UrlState serialization failed");
        urlencoding::encode(&json).to_string()
    }

    /// Deserialize from URL-safe base64 encoded JSON
    pub fn from_url_param(encoded: &str) -> Result<Self, String> {
        let decoded = urlencoding::decode(encoded)
            .map_err(|e| format!("URL decode failed: {}", e))?;
        let json = decoded.into_owned();
        serde_json::from_str(&json)
            .map_err(|e| format!("JSON parse failed: {}", e))
    }

    /// Check if state is default (no active filters/settings)
    pub fn is_default(&self) -> bool {
        self.view_mode == ViewMode::default()
            && !self.filter.is_active()
            && !self.heatmap.is_enabled()
    }
}

impl Default for UrlState {
    fn default() -> Self {
        Self {
            view_mode: ViewMode::default(),
            filter: BuildingFilter::default(),
            heatmap: DensityHeatmap::default(),
        }
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_url_state_serialization_roundtrip() {
        let original = UrlState {
            view_mode: ViewMode::TwoD,
            filter: BuildingFilter {
                min_year: 1950,
                ..Default::default()
            },
            heatmap: DensityHeatmap { enabled: true },
        };

        let encoded = original.to_url_param();
        let decoded = UrlState::from_url_param(&encoded).unwrap();

        assert_eq!(original.view_mode, decoded.view_mode);
        assert_eq!(original.filter.min_year, decoded.filter.min_year);
        assert_eq!(original.heatmap.enabled, decoded.heatmap.enabled);
    }

    #[test]
    fn test_url_state_default_no_encoding() {
        let state = UrlState::default();
        assert!(state.is_default());
    }
}
```

## Open Questions

1. **Dioxus 0.7 router query parameter support**
   - What we know: Codebase uses `#[route("/")]` macros, `Link` components
   - What's unclear: Whether `#[route("/path?param=value")]` or `use_route()` supports query params natively
   - Recommendation: Use JS interop `pushState()` pattern (proven in codebase) rather than experimenting with router

2. **Browser back button state restoration**
   - What we know: `popstate` event fires on back/forward
   - What's unclear: How to sync JS `popstate` to Rust signals in Dioxus WASM
   - Recommendation: Listen to `popstate` in JS, write to DOM element, read in Rust `use_effect`

3. **URL length limits with encoded state**
   - What we know: URLs have practical limits (~2000 chars in some browsers)
   - What's unclear: Whether serialized `UrlState` (filters + view + heatmap) exceeds this
   - Recommendation: Test with max filter values, implement compression if needed (e.g., msgpack instead of JSON)
