## 3. Context and Problem Statement

### 3.1 Problem Description

IOU-Modern requires AI-powered agents (Research, Content, Compliance, Review) to assist government employees with document creation, compliance checking, and Woo publication workflows. These agents need Large Language Model (LLM) capabilities with strong Dutch language support.

The initial architecture (ADR-005) specified OpenAI/Anthropic as the LLM provider. However, further analysis revealed concerns about **digital sovereignty**, **data residency**, **cost predictability**, and **vendor lock-in**. US Cloud Act could compel disclosure of Dutch government data processed by US-based AI providers.

**Problem statement as a question**: Which LLM provider strategy balances Dutch language capability, data sovereignty, cost control, and AI quality for government AI agents?

### 3.2 Why This Decision Is Needed

- **Business context**: BR-040 (AI compliance checking), BR-035-BR-045 (AI and Knowledge Graph features) require AI assistance
- **Technical context**: FR-023 to FR-028 (Knowledge Graph), FR-040 (AI compliance checking) define AI agent requirements
- **Regulatory context**: AVG (GDPR) requires EU data processing; RISK-STR-001 (Digital Sovereignty) identifies US provider risk

### 3.3 Supporting Links

- **Requirements**: BR-040, BR-041, BR-044, FR-023 to FR-028, FR-040, NFR-COMP-002 (AVG)
- **Risks**: RISK-STR-001 (Digital Sovereignty), RISK-STR-002 (Vendor Lock-In), RISK-FIN-002 (Ongoing Cost Overruns)
- **Architecture diagrams**: DIAG-002, DIAG-003, DIAG-004 (all updated with this decision)
- **Principles**: P4 (Sovereign Technology), P7 (Data Sovereignty)

## 4. Decision Drivers (Forces)

### 4.1 Technical Drivers

- **Dutch Language Quality**: Models must understand and generate Dutch text at government quality standards
  - Requirements: BR-040 (AI compliance checking), Woo publication workflow
  - Target: >90% accuracy on Dutch document classification

- **Integration Capability**: Must support REST API or gRPC integration with Rust backend
  - Requirements: FR-013 to FR-022 (Document Operations)
  - Architecture: Axum-based API with async AI agent orchestration

- **Response Time**: AI agents must return results within SLA
  - Requirements: NFR-PERF-003 (<500ms API response)
  - Impact: Synchronous vs. asynchronous AI processing

### 4.2 Business Drivers

- **Cost Predictability**: Usage-based pricing creates budget uncertainty
  - Requirements: RISK-FIN-002 (Operational Cost Overrun)
  - Target: Fixed or bounded monthly costs

- **Data Sovereignty**: Government data must remain under Dutch/EU jurisdiction
  - Requirements: RISK-STR-001 (Digital Sovereignty Compromise)
  - Policy: P7 (Data Sovereignty) principle

- **Vendor Independence**: Avoid lock-in to enable future provider switches
  - Requirements: RISK-STR-002 (Vendor Lock-In)
  - Target: Standard APIs, multiple provider options

### 4.3 Regulatory & Compliance Drivers

- **AVG/GDPR Compliance**: Personal data processing requires EU data residency
  - Article 45: Adequacy decisions for third-party transfers
  - Article 48: Cloud Act conflict with EU law
  - DPIA required: ARC-001-DPIA-v1.0

- **Woo Compliance**: AI-generated Woo assessments must be reviewable
  - BR-022: Human approval required for ALL Woo documents
  - BR-041: Human oversight for AI decisions

### 4.4 Alignment to Architecture Principles

| Principle | Alignment | Impact |
|-----------|-----------|--------|
| **P1: Privacy by Design** | ✅ Supports | EU-based processing ensures AVG compliance |
| **P4: Sovereign Technology** | ✅ Supports | Open source models enable independence from US providers |
| **P6: Human-in-the-Loop AI** | ✅ Supports | AI provides recommendations, humans decide |
| **P7: Data Sovereignty** | ✅ Supports | Mistral EU hosting; self-hosting for full control |

## 5. Considered Options

### Option 1: Mistral AI (Cloud API)

**Description**: Use Mistral AI's cloud API (Mistral Large, Mistral NeMo) hosted in EU regions.

**Implementation approach**:
- Integration via HTTPS API with Rust reqwest client
- Models: Mistral Large (complex reasoning), Mistral NeMo (fast Dutch responses)
- EU region hosting ensures data residency
- Fallback to self-hosted if service unavailable

**Wardley Evolution Stage**: Product (Mature commercial offering, moving toward Commodity)

#### Good (Pros)

- ✅ **EU Data Residency**: Mistral hosts in EU regions (Paris), complying with P7 principle
- ✅ **Dutch Language**: Strong Dutch language support in Mistral NeMo (trained on European languages)
- ✅ **Cost Effective**: ~70% cheaper than GPT-4 (€2/M tokens vs €7/M for competitors)
- ✅ **Open Source Roots**: Mistral releases models as open source, reducing lock-in risk
- ✅ **Quality**: Mistral Large approaches GPT-4 quality on many benchmarks
- ✅ **API Compatibility**: Standard OpenAI-compatible API simplifies integration

#### Bad (Cons)

- ❌ **Vendor Dependency**: Still dependent on Mistral's cloud service availability
- ❌ **Model Quality Gap**: Slightly below GPT-4/Claude Opus on complex reasoning tasks
- ❌ **Maturity**: Newer provider with less enterprise track record
- ❌ **Feature Set**: Fewer features than OpenAI/Anthropic (no advanced tool use yet)

#### Cost Analysis

- **CAPEX**: €5,000 (PoC, integration development, testing)
- **OPEX**: €8,000/month (estimated 100M tokens/month across all agents)
- **TCO (3-year)**: €293,000

|-------|--------|-------|
| 4. Open standards | Positive | OpenAI-compatible API, open source model availability |
| 5. Security | Positive | EU hosting, SOC2 certification |

### Option 2: Self-Hosted Open Source (Llama 3.x, Mistral Models)

**Description**: Self-host open source models (Llama 3.x 70B, Mistral 7B/8x7B) on government infrastructure.

**Implementation approach**:
- Models hosted on GPU servers in Netherlands government data center
- Inference via vLLM or TensorRT-LLM for performance
- API gateway provides OpenAI-compatible interface to agents
- Models: Llama 3.1 70B (reasoning), Mistral NeMo (Dutch speed)

**Wardley Evolution Stage**: Custom-Built (requires expertise and infrastructure)

#### Good (Pros)

- ✅ **Full Data Sovereignty**: Complete control over data, no third-party access
- ✅ **No API Costs**: Fixed hardware cost regardless of usage
- ✅ **Customization**: Models can be fine-tuned on government terminology
- ✅ **Unlimited Usage**: No rate limits or token quotas
- ✅ **Compliance**: Meets strictest Dutch government data requirements

#### Bad (Cons)

- ❌ **High CAPEX**: €100K-€200K for GPU servers (NVIDIA H100/A100 cluster)
- ❌ **Operational Overhead**: Requires ML engineering team for maintenance
- ❌ **Model Quality**: Open source models lag behind proprietary frontier models
- ❌ **Dutch Language**: Llama's Dutch is not as strong as Mistral NeMo
- ❌ **Scalability**: Must scale hardware manually for increased load

#### Cost Analysis

- **CAPEX**: €150,000 (GPU servers: 4x NVIDIA H100, networking, storage)
- **OPEX**: €36,000/year (hosting, maintenance, electricity, cooling)
- **TCO (3-year)**: €258,000

| Point | Impact | Notes |

| 4. Open standards | Positive | Full control, self-determined standards |
| 5. Security | Positive | On-premises, full audit trail |

### Option 4: Hybrid Approach (Mistral Cloud + Self-Hosted Fallback)

**Description**: Use Mistral cloud API as primary, with self-hosted Mistral models as fallback/burst.

**Implementation approach**:
- Primary: Mistral AI cloud API (EU region)

**Wardley Evolution Stage**: Product (Primary) + Custom-Built (Fallback)

- ✅ **Reliability**: Failover ensures high availability

- ✅ **Learning Path**: Build ML ops expertise while using cloud

## 6. Decision Outcome

### 6.1 Chosen Option

**Option 1: Mistral AI (Cloud API) with planned migration to Option 2 (Self-Hosted)**

**Primary Decision**: Use Mistral AI cloud API (Mistral Large, Mistral NeMo) for initial deployment.

**Strategic Path**: Plan migration to self-hosted open source models within 18-24 months as Llama/Mistral open source quality improves and ML ops expertise is built.

### 6.2 Y-Statement (Structured Justification)

> **In the context of** building AI-powered government document processing,
> **facing** data sovereignty requirements, cost constraints, and Dutch language needs,
> **we decided for** Mistral AI cloud API with a path to self-hosted open source,
> **to achieve** EU data residency, cost predictability, and vendor independence,
> **accepting** slightly lower model quality compared to GPT-4/Claude.

### 6.3 Justification (Why This Option?)

**Key reasons**:

1. **Data Sovereignty (P7 Principle)**: Mistral's EU hosting eliminates US Cloud Act risk. RISK-STR-001 (Digital Sovereignty) is reduced from MEDIUM to LOW.

2. **Cost Savings**: 70% reduction vs. OpenAI/Anthropic. RISK-FIN-002 (Ongoing Cost Overrun) impact reduced. 3-year TCO: €293K vs €902K for OpenAI/Anthropic.

3. **Vendor Independence**: Mistral's open source roots enable future self-hosting. RISK-STR-002 (Vendor Lock-In) mitigated.

4. **Dutch Language**: Mistral NeMo trained on European languages; quality tested as sufficient for Woo classification.

5. **Strategic Migration Path**: As Llama 4 and future Mistral models improve, self-hosting becomes viable. This decision buys time to build ML ops expertise.

**Stakeholder consensus**: Architecture Board approved with DPO support for EU data processing. Finance approved based on cost savings.

**Risk appetite**: MEDIUM - Accepting slightly lower model quality (85-90% of GPT-4) for sovereignty and cost benefits.

## 7. Consequences

### 7.1 Positive Consequences

- ✅ **Data Sovereignty Achieved**: EU-hosted processing eliminates US Cloud Act exposure
- ✅ **Cost Predictability**: 70% cost reduction vs. proprietary alternatives; budget certainty
- ✅ **Vendor Independence**: Open source compatibility enables future self-hosting
- ✅ **Compliance**: Meets P4 (Sovereign Technology) and P7 (Data Sovereignty) principles
- ✅ **Innovation**: Access to latest open source models as they're released

**Measurable outcomes**:

- AI Quality Score: 85% of GPT-4 baseline (acceptable for Woo classification)
- Monthly AI Cost: €8K (vs. €25K projected for OpenAI/Anthropic)
- Data Residency: 100% EU processing
- RISK-STR-001: Reduced from MEDIUM to LOW

### 7.2 Negative Consequences (Accepted Trade-offs)

- ❌ **Model Quality Gap**: Mistral Large ~85-90% of GPT-4 on complex reasoning
- ❌ **Feature Limitations**: No advanced tool use, vision, or multimodal capabilities yet
- ❌ **Maturity Risk**: Newer provider with less enterprise track record
- ❌ **Migration Effort**: Existing OpenAI/Anthropic integrations must be updated

**Mitigation strategies**:

- **Model Quality**: Human-in-the-loop (P6 principle) ensures quality despite lower AI scores
- **Feature Limitations**: Add features as Mistral releases them; use specialized vision models if needed
- **Maturity Risk**: Proof-of-concept completed before full rollout; SLA in contract
- **Migration**: DIAG-002, DIAG-003, DIAG-004 already updated

### 7.3 Neutral Consequences (Changes Needed)

- 🔄 **Team Training**: Developers learn Mistral API differences from OpenAI
- 🔄 **Contract Management**: New vendor agreement with Mistral AI
- 🔄 **Monitoring**: Track Mistral uptime, latency, quality metrics
- 🔄 **Fallback Plan**: Document procedures for Mistral outages

**Break-even**: Self-hosted becomes cheaper at ~2.5M tokens/month usage

- Build ML ops expertise
- Collect training data (Dutch government documents)
- Evaluate model quality requirements

**Phase 2 (Months 12-18)**: Hybrid approach
