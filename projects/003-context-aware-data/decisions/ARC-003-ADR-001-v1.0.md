## Context

### Background

The Context-Aware Data Architecture requires a database system capable of:

1. **Storing contextual metadata** for millions of government information objects
2. **Maintaining referential integrity** between information objects and their context
3. **Supporting complex queries** across multiple context layers
4. **Enabling semantic similarity search** for context discovery
5. **Complying with AVG/GDPR** requirements for government data
6. **Integrating with existing MinJus infrastructure** and standards

### Problem Statement

The Ministry of Justice & Security needs a database solution that balances:
- **Strong consistency** for context integrity (principle C5)
- **Flexible schema** for evolving context types (principle C17)
- **Advanced search** capabilities including vector similarity
- **Government compliance** (AVG, Archiefwet, Woo)
- **Operational maturity** with proven government deployments

### Drivers

**Technical Drivers**:
- ACID guarantees required for referential integrity
- JSON support needed for flexible context_value storage
- Full-text search required for context discovery
- Vector similarity needed for semantic context matching

**Business Drivers**:
- AVG compliance with EU data residency
- Proven government deployments
- Integration with existing PostgreSQL infrastructure
- Cost containment

**Constraint Drivers**:
- Must support Dutch language text processing
- Must integrate with existing MinJus identity provider
- Must support 20-year retention requirements

## Decision

### Decision Statement

**PostgreSQL 15+ with pgvector extension is selected as the primary database for the Context-Aware Data system.**

### Key Components

| Component | Version | Purpose |
|-----------|---------|---------|
| **PostgreSQL** | 15+ | Core database engine |
| **pgvector** | 0.5.0+ | Vector similarity search |
| **apache_age** | 1.3.0+ | Graph queries for provenance |
| **Flyway** | 9.0+ | Database migrations |

### Schema Strategy

**Relational Tables** (Core entities):
- `informationobject` - Primary data entities
- `context` - Contextual metadata records
- `contexttype` - Type definitions
- `contextlayer` - Layer definitions

## Alternatives Considered

### Alternative 1: MongoDB (NoSQL Document Store)

**Description**: Use MongoDB as a document-oriented database for flexible context storage.

**Pros**:
- Flexible schema for evolving context types
- Built-in sharding for horizontal scaling
- Native JSON support
- Good for nested document structures

**Cons**:
- Weak referential integrity (violates principle C5)
- No ACID guarantees across documents
- Limited full-text search capabilities
- No native vector similarity support
- Less proven in Dutch government

**Rejected Because**: Weak referential integrity is unacceptable for context-data relationships (principle C5: Context-Integrity).

### Alternative 2: ArangoDB (Multi-Model Database)

**Description**: Use ArangoDB for both document and graph capabilities.

**Pros**:
- Native graph support for provenance queries
- Flexible document model
- Built-in query language (AQL)
- Good for hybrid workloads

**Cons**:
- Less mature than PostgreSQL
- Smaller talent pool
- Limited government deployments in NL
- Less extensive ecosystem
- Higher operational complexity

**Rejected Because**: Operational complexity and limited government adoption pose unnecessary risk.

### Alternative 3: Oracle RDBMS

**Description**: Use Oracle as the enterprise database solution.

**Pros**:
- Proven enterprise reliability
- Strong ACID guarantees
- Advanced features (JSON, spatial, etc.)
- Existing Oracle expertise in MinJus

**Cons**:
- Very high licensing costs
- Complex licensing for vector extensions
- Heavier infrastructure requirements
- Slower innovation cycle

**Rejected Because**: High cost and complexity without proportional benefit over PostgreSQL.

### Alternative 4: Microsoft SQL Server

**Description**: Use SQL Server as the database solution.

**Pros**:
- Strong enterprise features
- Good JSON support
- Proven reliability
- Existing MS expertise in MinJus

**Cons**:
- Licensing costs
- Limited vector support
- Windows-centric (vs Linux preference)
- Less flexible extension ecosystem

**Rejected Because**: PostgreSQL offers equivalent capabilities at lower cost with better extensibility.

## Consequences

### Positive Consequences

**Technical Benefits**:
1. **Strong ACID guarantees** ensure context integrity (addresses C5)
2. **pgvector extension** enables semantic similarity search
3. **JSON support** provides flexibility for evolving context types (addresses C17)
4. **Full-text search** built-in for context discovery
5. **Mature ecosystem** with extensive tooling and expertise
6. **AVG-compliant** with EU data residency options

**Operational Benefits**:
1. **Lower TCO** compared to commercial alternatives
2. **Existing PostgreSQL expertise** in MinJus
3. **Proven reliability** in government deployments
4. **Strong community support** and long-term viability
5. **Easy integration** with existing MinJus infrastructure

**Business Benefits**:
1. **Faster time-to-market** with familiar technology
2. **Lower licensing costs** (€0 vs €100k+ for Oracle/SQL Server)
3. **Reduced vendor lock-in** with open source
4. **Better talent availability** for PostgreSQL skills

### Negative Consequences

**Technical Limitations**:
1. **Horizontal scaling** requires manual sharding (vs automatic in MongoDB)
2. **Vector operations** may be slower than dedicated vector databases
3. **JSON query performance** lower than dedicated document stores
4. **Graph queries** limited compared to native graph databases

**Operational Challenges**:
1. **Requires DBA expertise** for PostgreSQL optimization
2. **Extension management** adds complexity
3. **Backup/restore** complexity for large datasets
4. **Connection pooling** requires careful configuration

**Mitigation Strategies**:
- Use read replicas to address scaling (sufficient for 10M records)
- Implement caching layer (Redis) for frequently accessed context
- Regular performance testing and optimization
- DBA training and external support contracts

### Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Performance at scale | MEDIUM | HIGH | Early load testing, read replicas, caching |
| pgvector limitations | LOW | MEDIUM | Hybrid approach with specialized vector DB if needed |
| Extension compatibility | LOW | MEDIUM | Version pinning, testing before upgrades |
| Talent availability | LOW | LOW | Training, external support, partner ecosystem |

## Compliance Considerations

### AVG/GDPR Compliance

**Data Protection**:
- Encryption at rest: AES-256 (PostgreSQL native)
- Encryption in transit: TLS 1.3
- Row-level security for PII context
- Audit logging for all access

**Data Subject Rights**:
- Right to access: Queryable via API
- Right to rectification: Update capabilities
- Right to erasure: Anonymization after retention
- Right to portability: JSON export functionality

**DPIA Status**: Required - addressed in project documentation

### Archiefwet 1995 Compliance

**Retention**:
- 20-year retention for decisions
- Automated retention calculation
- Archive preparation workflows

**Integrity**:
- Referential integrity enforced
- Audit trail maintained
- Versioning supported

### Government Standards

**Compliant With**:
- NOPOD (Nederlandse Openstandarden) principles
- Netherlands AI and Human Rights read
- Digital Government Act requirements
