## Context

### Background

The Context-Aware Data Architecture aims to capture contextual metadata for all government information objects. However, not all context can be practically captured at source (principle C6: Capture at Source):

**Manual Capture Challenges**:
- Legacy systems without context fields
- Historical documents missing context
- Complex context requiring expert analysis
- Resource constraints for manual entry
- Specialized context (legal references, subjects)

**Principle C7** states: *"When context cannot be captured at source, systems MAY use automated inference, but MUST clearly distinguish between captured and inferred context."*

### Problem Statement

The architecture needs a solution for:

1. **Context Discovery**: Automatically identify context from document content
2. **Legal References**: Extract BWBR article references from decisions
3. **Subject Classification**: Categorize documents by policy domain
4. **Entity Recognition**: Identify persons, organizations, locations
5. **Quality Assurance**: Ensure inferred context is reliable and valid
6. **Transparency**: Clearly distinguish captured vs. inferred context
7. **Human Oversight**: Enable validation for high-stakes inferences

### Requirements

**Functional Requirements**:
- FR-I-001: Infer legal basis from decision text
- FR-I-002: Classify documents by subject domain
- FR-I-003: Extract named entities (persons, orgs, cases)
- FR-I-004: Provide confidence scores for all inferences
- FR-I-005: Support human review and correction

**Non-Functional Requirements**:
- NFR-I-001: > 0.8 confidence for auto-accepted inferences
- NFR-I-002: < 5 second inference latency
- NFR-I-003: 95%+ accuracy on legal reference extraction
- NFR-I-004: Full audit trail for all inferences

**Compliance Requirements**:
- CR-I-001: AVG-compliant processing (DPIA required)
- CR-I-002: Human review for high-risk inferences
- CR-I-003: Transparency about inference methods
- CR-I-004: Right to correction for inferred context

## Decision

### Decision Statement

**Implement a Context Inference Service using Anthropic Claude API for LLM-based inference combined with spaCy for NER, with mandatory human review for confidence scores below 0.9 and all legal context inferences.**

## Alternatives Considered

### Alternative 1: Self-Hosted Open Source LLM

**Description**: Deploy open-source LLM (e.g., Llama 2, Mistral) on MinJus infrastructure.

**Pros**:
- No external API dependency
- Data stays on-premises (privacy advantage)
- Potentially lower cost at scale
- Custom fine-tuning possible

**Cons**:
- ❌ High infrastructure cost (GPU servers)
- ❌ Model maintenance burden
- ❌ Lower quality than frontier models
- ❌ Requires specialized ML ops team
- ❌ Higher latency (on-prem vs API)

**Rejected Because**: The quality gap with frontier models (Claude, GPT-4) is significant for legal context. Infrastructure costs for self-hosting are substantial. The operational burden is beyond current team capabilities.

### Alternative 2: Rule-Based Inference Only

**Description**: Use only deterministic rules and regex patterns, no AI/ML.

**Pros**:
- Fully predictable and explainable
- No API costs
- Fast execution
- No ML expertise needed

**Cons**:
- ❌ Cannot handle semantic complexity
- ❌ Brittle (breaks on document variations)
- ❌ Limited to predefined patterns
- ❌ Poor performance on legal references
- ❌ Cannot improve over time

**Rejected Because**: Rule-based systems are insufficient for extracting semantic meaning like legal basis or subject classification from varied government documents. The error rate would be unacceptably high.

### Alternative 3: Human-Only Context Capture

**Description**: All context manually entered by domain experts, no automation.

**Pros**:
- 100% accuracy (when experts agree)
- Full control and transparency
- No AI complexity
- No privacy concerns with AI

**Cons**:
- ❌ Not scalable (expert time limited)
- ❌ Inconsistent (human variation)
- ❌ Expensive (expert labor cost)
- ❌ Slow (manual process)
- ❌ Incomplete (historical documents)

**Rejected Because**: Manual-only approach violates principle C2 (Minimal Overhead) and is not scalable to millions of documents. The operational cost would be prohibitive.

### Alternative 4: Google Cloud AI / Azure OpenAI

**Description**: Use cloud provider AI services instead of Anthropic Claude.

**Pros**:
- Multiple model options
- Integration with cloud ecosystem
- Potential volume discounts

**Cons**:
- ❌ Data residency concerns (US cloud)
- ❌ Model quality varies
- ❌ Vendor lock-in risk
- ❌ Less transparent than Anthropic on safety

**Rejected Because**: Anthropic Claude has demonstrated superior performance on legal/government text with strong safety guarantees. The Constitutional AI approach aligns with government values.

## Consequences

### Positive Consequences

**Quality**:
1. ✅ **High accuracy**: Claude achieves > 0.9 on legal reference extraction
2. ✅ **Continuous improvement**: Model updates improve over time
3. ✅ **Consistent**: Eliminates human variation

**Efficiency**:
1. ✅ **Scalable**: Can process millions of documents
2. ✅ **Fast**: < 5 seconds per document
3. ✅ **Cost-effective**: API pricing vs. expert labor

**Compliance**:
1. ✅ **Transparent**: All inferences marked and traceable
2. ✅ **Reviewable**: Human validation for high-risk cases
3. ✅ **Correctable**: Inferred context can be overridden

**Capability**:
1. ✅ **Semantic understanding**: Extracts meaning, not just patterns
2. ✅ **Legal expertise**: Strong performance on legal text
3. ✅ **Multilingual**: Handles Dutch government documents

### Negative Consequences

**Dependency Risks**:
1. ⚠️ **API dependency**: Service interruption halts inference
2. ⚠️ **Cost variability**: Usage-based pricing can fluctuate
3. ⚠️ **Model changes**: Provider updates may affect behavior

**Privacy Concerns**:
1. ⚠️ **Data sent externally**: Documents processed by Anthropic
2. ⚠️ **Training data use**: Must verify no training on inputs
3. ⚠️ **DPIA complexity**: External AI processing requires careful analysis

**Quality Risks**:
1. ⚠️ **Hallucination risk**: LLM may invent context
2. ⚠️ **Cultural bias**: Model may not understand NL context
3. ⚠️ **Legal precision**: May misinterpret legal references

### Mitigation Strategies

**Dependency Mitigations**:
- Implement fallback to rule-based inference
- Cache inferences to avoid re-processing
- Monitor API costs with alerts
- Contract review for SLA guarantees

**Privacy Mitigations**:
- Anonymize PII before sending to API
- Contractual guarantee: no training on inputs
- Data processing agreement in place
- Evaluate on-prem option for Phase 2

**Quality Mitigations**:
- Mandatory human review for legal context
- Confidence threshold (0.9) for auto-accept
- Regular accuracy audits
- Feedback loop for corrections

## Privacy and Compliance

### DPIA Requirements

**High-Risk Processing**: AI inference on government data triggers DPIA requirement.

**Risk Assessment**:

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| PII in inference output | MEDIUM | HIGH | Anonymization, access controls |
| Incorrect legal references | MEDIUM | HIGH | Human review, validation |
| Training on inputs | LOW | HIGH | Contractual prohibition |
| External processing | MEDIUM | MEDIUM | Data processing agreement |

**DPIA Status**: Required (in progress, target Q2 2026)

### AVG/GDPR Considerations

**Legal Basis**:
- Automated processing: Art 6(1)(c) Legal Obligation
- Special category data: Art 9(2)(g) Substantial public interest

**Data Subject Rights**:
- Right to explanation: Inference method documented
- Right to correction: Inferred context can be overridden
- Right to object: Opt-out of inference for personal data

**Transparency Requirements**:
- Clearly mark inferred context in UI
- Provide confidence scores
- Explain inference method on request
- Maintain audit trail

## Human Review Workflow

### Review Triggers

Inference requires human review if:
1. Confidence score < 0.9
2. Inference type is "legal_basis" (always review)
3. Inference type is "woo_classification" (always review)
4. Document is marked as high-importance
5. Previous corrections on similar documents

### Review Process

```
┌─────────────────────────────────────────────────────────────────┐
│                     HUMAN REVIEW WORKFLOW                       │
│                                                                  │
│  1. Steward receives review notification                         │
│  2. Review UI shows:                                            │
│     - Document text                                              │
│     - Inferred context + confidence                              │
│     - Source evidence (extracted text)                          │
│  3. Steward decides:                                            │
│     - APPROVE: Context accepted as-is                           │
│     - MODIFY: Adjust context value                              │
│     - REJECT: Remove inference entirely                         │
│  4. Decision recorded and stored                                 │
│  5. Feedback used for model improvement                          │
└─────────────────────────────────────────────────────────────────┘
```

### Review SLAs

| Priority | Response Time | Review Target |
|----------|---------------|---------------|
| Critical (legal) | 4 hours | Same day |
| High | 1 day | 2 days |
| Normal | 3 days | 1 week |

## Rollback Plan

If inference quality proves insufficient:

**Phase 1**: Increase human review threshold (0.9 → 0.95)
**Phase 2**: Disable automatic acceptance for legal context
**Phase 3**: Fall back to rule-based for high-volume types
**Phase 4**: Revert to manual-only if quality unacceptable

**Triggers for Re-evaluation**:
- Correction rate exceeds 10%
- Legal reference accuracy drops below 0.9
- Significant errors discovered in audit

**Timeline for Re-evaluation**: Quarterly
