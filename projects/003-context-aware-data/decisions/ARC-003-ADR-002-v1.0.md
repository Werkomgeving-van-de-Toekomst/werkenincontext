## Context

### Background

The Context-Aware Data Architecture must store contextual metadata for government information objects. A fundamental architectural question is whether to store context:

1. **Co-located** with the parent information object (same database, same transaction)
2. **Separately** in a dedicated context service (different database, eventual consistency)

This decision impacts data integrity, query performance, system complexity, and compliance with architecture principles.

### Problem Statement

The architecture must balance competing requirements:

**Integrity Requirements**:
- Principle C5 (Context-Integrity): Context must maintain referential integrity with parent
- Archiefwet: Context must archive atomically with parent document
- Woo queries: Context must be consistently available with document

**Flexibility Requirements**:
- Principle C17 (Context Evolution): Context types evolve independently
- Multiple source systems may contribute context for same object
- Context may be added after initial object creation

**Performance Requirements**:
- < 500ms API response time for object+context retrieval
- Support for high query volume (Woo searches)
- Efficient context-only queries (quality monitoring, analytics)

### Principles Alignment

This decision directly addresses multiple architecture principles:

| Principle | Requirement | Implication |
|-----------|-------------|-------------|
| **C5: Context-Integrity** | Referential integrity enforced | Co-location strongly preferred |
| **C8: Context Co-Location** | Explicitly stated in principles | This decision validates C8 |
| **C6: Capture at Source** | Context captured with object | Supports co-location |
| **C12: Context Validation** | Validation requires object access | Easier with co-location |
| **C13: Context Freshness** | Consistency between object+context | Co-location ensures consistency |

## Decision

### Decision Statement

**Contextual metadata MUST be stored co-located with its parent information object in the same PostgreSQL database, ensuring atomic access, referential integrity, and transactional consistency.**

### Architecture Pattern

```
┌─────────────────────────────────────────────────────────────────┐
│                    CO-LOCATED STORAGE MODEL                      │
│                                                                  │
│  InformationObject Table           Context Table                 │
│  ┌─────────────────────────┐       ┌─────────────────────────┐ │
│  │ object_id (PK)          │───┐   │ context_id (PK)         │ │
│  │ title                   │   │   │ object_id (FK) ─────────│─│
│  │ object_type             │   │   │ layer_id (FK)          │ │
│  │ domain_id               │   │   │ type_id (FK)           │ │
│  │ created_at              │   │   │ context_value          │ │
│  │ created_by              │   │   │ captured_at            │ │
│  └─────────────────────────┘   │   └─────────────────────────┘ │
│                                │                                │
│  Same Database              Same Transaction                 │
│  (PostgreSQL)              (Atomic Access)                   │
└─────────────────────────────────────────────────────────────────┘
```

### Implementation Strategy

**Database Schema**:
- Context stored in `context` table with foreign key to `informationobject`
- Foreign key constraint with CASCADE delete for orphan prevention
- Same database cluster for all context-aware data

**API Design**:
- Single API endpoint returns object + context atomically
- Context creation/update in same transaction as object operations
- Context queries join directly with object table

**Transaction Management**:
```sql
BEGIN;
-- Insert or update information object
INSERT INTO informationobject (...) VALUES (...);
-- Create context in same transaction
INSERT INTO context (object_id, ...) VALUES (...);
COMMIT;
```

### Access Patterns

**Primary Pattern: Object with Context**
```sql
SELECT io.*, 
       c.layer_name, ct.type_name, ctx.context_value
FROM informationobject io
LEFT JOIN context ctx ON ctx.object_id = io.object_id
JOIN contexttype ct ON ct.type_id = ctx.type_id
JOIN contextlayer c ON c.layer_id = ct.layer_id
WHERE io.object_id = ?;
```

**Secondary Pattern: Context-Only Queries**
```sql
SELECT ctx.*, io.title
FROM context ctx
JOIN informationobject io ON io.object_id = ctx.object_id
WHERE ctx.type_id = ? AND ctx.context_value ILIKE ?;
```

## Alternatives Considered

### Alternative 1: Separate Context Service (Microservices)

**Description**: Store context in a separate microservice with its own database, accessed via API.

**Pros**:
- Independent scaling of context service
- Technology flexibility (different database per service)
- Clear service boundaries
- Team autonomy

**Cons**:
- ❌ Violates referential integrity (distributed transactions)
- ❌ Eventual consistency delays context availability
- ❌ Complex orchestration for atomic operations
- ❌ Network latency for context access
- ❌ Violates principle C8 (Context Co-Location)
- ❌ Archival complexity (must coordinate transfers)
- ❌ Doubles infrastructure cost

**Rejected Because**: Distributed transactions across services are complex and fragile. The principle C8 explicitly requires co-location, and the integrity requirements (C5) cannot be reliably met with distributed state.

### Alternative 2: Hybrid Approach (Co-Located + Cached)

**Description**: Store context co-located but maintain a Redis cache for frequently accessed context.

**Pros**:
- Retains integrity (primary store co-located)
- Improves read performance via cache
- Supports high query volume

**Cons**:
- Added complexity (cache invalidation)
- Stale cache risk
- Additional infrastructure
- Cache warming required

**Rejected Because**: Cache adds complexity without proportional benefit. PostgreSQL performance with proper indexing is sufficient. Cache can be added later if needed (you don't pay it forward).

### Alternative 3: Context Embedded in Object (JSON Column)

**Description**: Store context as JSONB within the informationobject table.

**Pros**:
- Simplest access pattern (single table)
- No joins required
- Natural document structure

**Cons**:
- ❌ No referential integrity for context types
- ❌ Cannot query context efficiently
- ❌ Cannot enforce validation rules at database level
- ❌ Context quality monitoring difficult
- ❌ Violates principle C3 (Explicit Context Model)

**Rejected Because**: Embedding context in JSON sacrifices the explicit context model required by principle C3. It makes context validation, quality monitoring, and governance significantly harder.

### Alternative 4: Event-Sourced Context

**Description**: Store context as an event log, reconstructing current state via event replay.

**Pros**:
- Complete audit trail natively
- Temporal queries (context at any point in time)
- Event-driven architecture alignment

**Cons**:
- ❌ Complex query patterns (requires replay)
- ❌ Performance overhead for current state queries
- ❌ Steep learning curve for developers
- ❌ Limited tooling support

**Rejected Because**: Event sourcing adds complexity that is not justified for this use case. Provenance is important but can be achieved with simpler audit logging.

## Consequences

### Positive Consequences

**Data Integrity**:
1. ✅ **Atomic transactions**: Object and context updated together
2. ✅ **Referential integrity**: Foreign key constraints enforced
3. ✅ **No orphaned context**: CASCADE deletes prevent orphans
4. ✅ **Archival atomicity**: Object + context archive together

**Query Performance**:
1. ✅ **Single database queries**: No network latency
2. ✅ **Join optimization**: PostgreSQL optimizer works well
3. ✅ **Efficient context queries**: Proper indexing on foreign keys
4. ✅ **Full-text search**: Native PostgreSQL capabilities

**Operational Simplicity**:
1. ✅ **Single database cluster**: Reduced infrastructure
2. ✅ **Simple backup/restore**: One database to manage
3. ✅ **Straightforward migrations**: Coordinated schema changes
4. ✅ **Easier troubleshooting**: No distributed transactions

**Compliance**:
1. ✅ **AVG compliance**: Single point of access control
2. ✅ **Archiefwet compliance**: Atomic archival
3. ✅ **Audit trail**: Simple to implement with triggers

**Principle Alignment**:
1. ✅ **C5: Context-Integrity** - Enforced by foreign keys
2. ✅ **C8: Context Co-Location** - Explicitly implemented
3. ✅ **C12: Context Validation** - Database constraints enforce rules
4. ✅ **C13: Context Freshness** - No consistency lag

### Negative Consequences

**Scalability Constraints**:
1. ⚠️ **Single database scaling**: Must scale entire database together
2. ⚠️ **Potential write bottleneck**: All writes go to same cluster
3. ⚠️ **Large database size**: Context increases storage requirements

**Mitigation Strategies**:
- Use read replicas for read scaling
- Partition tables by object type or date
- Archive old data to cold storage
- Connection pooling for high concurrency

**Development Complexity**:
1. ⚠️ **Schema coordination**: Object and context schema changes coordinated
2. ⚠️ **Migration complexity**: Larger schema changes affect more tables

**Mitigation Strategies**:
- Use Flyway for version-controlled migrations
- Additive changes only where possible
- Use views for backward compatibility
- Automated testing for schema changes

### Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Database becomes bottleneck | MEDIUM | HIGH | Read replicas, caching, partitioning |
| Schema change coordination | MEDIUM | MEDIUM | Migration framework, additive changes |
| Large table performance | LOW | MEDIUM | Partitioning, archiving, indexing |
| Single point of failure | LOW | HIGH | High availability setup, replication |

## Implementation Guidance

### Database Schema

**Core Tables**:
```sql
-- Information object (parent entity)
CREATE TABLE informationobject (
    object_id UUID PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    object_type ENUM NOT NULL,
    domain_id UUID NOT NULL,
    created_at TIMESTAMP NOT NULL,
    created_by UUID NOT NULL,
    -- ... other fields
);

-- Context (co-located child)
CREATE TABLE context (
    context_id UUID PRIMARY KEY,
    object_id UUID NOT NULL,
    layer_id UUID NOT NULL,
    type_id UUID NOT NULL,
    context_value TEXT,
    captured_at TIMESTAMP NOT NULL,
    FOREIGN KEY (object_id) REFERENCES informationobject(object_id) 
        ON DELETE CASCADE,
    FOREIGN KEY (layer_id) REFERENCES contextlayer(layer_id),
    FOREIGN KEY (type_id) REFERENCES contexttype(type_id)
);

-- Index for efficient queries
CREATE INDEX idx_context_object ON context(object_id);
CREATE INDEX idx_context_type ON context(type_id);
CREATE INDEX idx_context_layer ON context(layer_id);
```

### API Transaction Pattern

```java
@Transactional
public InformationObject createContextWithContext(
    CreateObjectRequest request
) {
    // 1. Create information object
    InformationObject obj = informationObjectRepository.save(request.toObject());
    
    // 2. Create context records
    for (ContextValue ctx : request.getContext()) {
        ctx.setObjectId(obj.getId());
        contextRepository.save(ctx);
    }
    
    // 3. Validate quality
    qualityService.validate(obj.getId());
    
    return obj;
}
```

### Query Patterns

**Retrieve Object with All Context**:
```java
@Query("""
    SELECT io, ctx 
    FROM InformationObject io 
    LEFT JOIN FETCH io.contexts ctx
    WHERE io.id = :id
""")
InformationObject findAllWithContext(@Param("id") UUID id);
```

**Quality Monitoring Query**:
```sql
SELECT 
    c.layer_name,
    ct.is_required,
    COUNT(*) FILTER (WHERE ctx.context_value IS NOT NULL) as populated,
    COUNT(*) as total,
    ROUND(
        100.0 * COUNT(*) FILTER (WHERE ctx.context_value IS NOT NULL) / COUNT(*),
        2
    ) as completeness_pct
FROM context ctx
JOIN contexttype ct ON ct.type_id = ctx.type_id
JOIN contextlayer c ON c.layer_id = ct.layer_id
GROUP BY c.layer_name, ct.is_required;
```

## Performance Considerations

### Expected Volume

| Metric | Value | Notes |
|--------|-------|-------|
| Initial objects | 1M | Estimated from existing systems |
| Growth rate | 50k/month | New objects + context |
| Peak volume | 3M | Year 5 projection |
| Avg context per object | 5 | Mix of core/domain/semantic/provenance |
| Total context records | 15M | At Year 5 |

### Query Performance

**Single Object Query** (with all context):
- Expected rows: 1 object + 5 context
- Target time: < 100ms
- Optimization: Covered indexes on foreign keys

**Context-Only Queries** (quality monitoring):
- Expected rows: 15M (full table scan)
- Target time: < 500ms
- Optimization: Partial indexes, materialized views

### Scaling Strategy

**Read Scaling**:
- Primary for writes
- 2-3 read replicas for queries
- Connection pooling (HikariCP)

**Write Scaling**:
- Single primary sufficient for 50k writes/month
- Partitioning by created_at (monthly)
- Archive old data to cold storage

## Rollback Plan

If co-location proves problematic:

**Phase 1**: Add caching layer for context reads
**Phase 2**: Implement read replicas for scaling
**Phase 3**: Consider database sharding by domain

**Trigger for Re-evaluation**:
- Database performance degrades despite optimization
- Write throughput exceeds single database capacity
- Query latency consistently exceeds SLAs

**Timeline for Re-evaluation**: 12 months post-implementation
