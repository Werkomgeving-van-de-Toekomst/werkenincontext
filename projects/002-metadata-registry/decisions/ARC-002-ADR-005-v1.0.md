## Context

The Dutch Government's "Sovereign Technology" principle requires that government IT systems use open-source technologies that ensure digital sovereignty. This principle is part of the Netherlands' digital strategy to avoid vendor lock-in, ensure security, and maintain control over critical government infrastructure.

### Sovereign Technology Criteria

1. **Open Source**: Software must be licensed under OSI-approved licenses
2. **EU Jurisdiction**: Data must be stored within EU/EEA
3. **No Vendor Lock-in**: Must be able to switch providers without major rework
4. **Community Governance**: Active community and meritocratic governance
5. **Auditability**: Source code must be available for security audits

### Constraints

- **BC-4**: Budget limited to open-source solutions (no commercial database licenses)
- **BR-MREG-017**: All data must be stored within EU jurisdiction
- **BR-MREG-023**: Must comply with Sovereign Technology principle
- **Government Policy**: Forum Standaardisatie mandates open-source preference

### Current State

IOU-Modern has already adopted Rust and ArangoDB, both compliant with Sovereign Technology. This decision formalizes the principle for the Metadata Registry Service.

## Decision

**Selected Option**: Full Sovereign Technology Compliance

**The Metadata Registry Service will use 100% open-source technologies with EU-based deployment and active community governance.**

### US Cloud Services (RESTRICTED)

| Service | Company | Restriction |
|---------|---------|-------------|
| AWS DynamoDB | Amazon (US) | Only for non-sensitive data |
| Azure Cosmos DB | Microsoft (US) | Only for non-sensitive data |
| Google Firestore | Google (US) | Only for non-sensitive data |

**Note**: For sensitive government data, EU-based alternatives are required:
- **AWS**: AWS Europe (Frankfurt) regions with EU data residency
- **Azure**: Azure Germany (separate cloud)
- **Google**: Google Cloud Brussels region

## Compliance Verification

### Open Source License Audit

```bash
# Run cargo-license audit
cargo install cargo-license
cargo license > licenses.txt

# Validate OSI-approved licenses
cargo deny check licenses

# Check for CVE vulnerabilities
cargo audit
```

### License Summary

```txt
cargo license --json | jq '.dependencies | group_by(.license) | map({
    license: .[0].license,
    count: length
})'
```

**Expected Result**: 100% OSI-approved licenses (MIT, Apache-2.0, BSD-2/3-Clause, MPL-2.0)

### Data Residency Verification

```yaml
# infrastructure/terraform/main.tf
resource "aws_kms_key" "arangodb_encryption" {
  provider = aws.eu-central-1

  description = "ArangoDB encryption key (EU region)"

  provider = aws.eu-central-1

  # EU data residency requirement
  enabled_cloudwatch_logs_exports = ["audit"]

  # Server-side encryption with EU-managed key
  kms_key_arn = aws_kms_key.arangodb_encryption.arn
}
```

## Vendor Lock-in Prevention

### Containerization

```dockerfile
# All services containerized for portability
FROM rust:1.75-alpine AS builder
WORKDIR /app
COPY Cargo.* ./
RUN cargo fetch --locked
COPY . .
RUN cargo build --release --frozen

FROM alpine:3.19
RUN apk add --no-cache ca-certificates
COPY --from=builder /app/target/release/metadata-api /usr/local/bin/
EXPOSE 8080
CMD ["metadata-api"]
```

### Kubernetes Deployment

```yaml
# infrastructure/kubernetes/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: metadata-registry
  namespace: metadata-registry
spec:
  replicas: 3
  selector:
    matchLabels:
      app: metadata-registry
```

## Governance

### Annual Compliance Review

**Date**: Q1 each year
**Participants**: Enterprise Architect, CIO Office, Legal/Compliance

**Review Checklist**:
- [ ] All dependencies have OSI-approved licenses
- [ ] No critical CVE vulnerabilities unpatched
- [ ] Data stored in EU/EEA regions
- [ ] No vendor lock-in risks identified
- [ ] Active community maintenance for all dependencies
- [ ] Documentation available for compliance audits

### Compliance Exceptions

**Process**: Any exception to Sovereign Technology principle requires:
1. Business justification (documented)
2. CIO approval
3. Legal/Compliance review
4. Mitigation plan
5. Annual renewal

**Current Exceptions**: None
