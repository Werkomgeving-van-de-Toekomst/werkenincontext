## Context

The Metadata Registry Service requires an admin UI for:
- **Metadata stewards** to manage GGHH entities
- **Domain owners** to approve changes and Woo publications
- **DPO/compliance officers** to review audit logs and SAR requests
- **Woo officers** to manage publication workflows

### Constraints

- **TC-3**: Must compile to WebAssembly (browser-based)
- **TC-4**: Must use Rust for type safety across stack
- **NFR-MREG-U-1**: Must be usable by non-technical domain owners
- **NFR-MREG-P-1**: UI interactions must feel instant (<100ms)

### Current State

The existing IOU-Modern codebase has no established frontend framework. The Metadata Registry admin UI is currently a stub implementation.

---

## Decision

**Selected Option**: Dioxus 0.7 (WebAssembly)

**Dioxus will be used for the admin UI, compiled to WebAssembly and served as a single-page application.**

---

## Alternatives Considered

### Option 1: Dioxus (SELECTED)

**Pros**:
- Rust-native (shared types with backend)
- Compiles to WebAssembly for performance
- React-like component model (familiar to many devs)
- Single codebase language (Rust only)
- Strong typing eliminates UI bugs at compile time
- Small bundle size (~100KB gzipped)
- Open-source (MIT/Apache 2.0)

**Cons**:
- Smaller ecosystem compared to React/Vue
- Fewer UI component libraries
- Steeper learning curve for frontend devs
- Less mature tooling (debugging, HMR)
- Limited CSS-in-JS options

**Impact**: Type-safe, performant UI aligned with Rust backend

---

### Option 2: React + TypeScript

**Pros**:
- Largest ecosystem and component libraries
- Excellent tooling (Vite, React DevTools)
- Huge talent pool
- Best-in-class HMR and DX
- Proven at scale

**Cons**:
- Separate TypeScript codebase from Rust backend
- Type definitions must be maintained separately
- Larger bundle size (200-500KB typical)
- Runtime type safety only (types erased)
- Cannot share business logic with backend

**Impact**: Best developer experience but maintenance overhead for type sharing

---

### Option 3: Vue 3 + TypeScript

**Pros**:
- Gentle learning curve
- Excellent documentation
- Strong performance
- Good ecosystem

**Cons**:
- Separate TypeScript codebase
- Type definitions maintenance
- Smaller talent pool than React
- Less mature WebAssembly support

**Impact**: Good balance but similar type-sharing challenges as React

---

### Option 4: Leptos (Rust/WASM)

**Pros**:
- Rust-native like Dioxus
- Fine-grained reactivity (signals)
- Excellent performance
- Strong typing

**Cons**:
- Less mature than Dioxus
- Smaller community
- Steeper learning curve
- Fewer examples and templates

**Impact**: Emerging framework with higher risk

## Consequences

### Positive Consequences

1. **Type Sharing**: Entities, validation rules shared with backend via `metadata-core`
2. **Performance**: WebAssembly provides native-like performance
3. **Bundle Size**: ~100KB gzipped vs 200-500KB for typical React app
4. **Single Language**: Full-stack Rust reduces context switching
5. **Sovereign Technology**: Fully open-source stack

### Negative Consequences

1. **Limited Components**: Fewer pre-built UI components
2. **Styling**: Noж€ђз†џзљ„ CSS-in-JS solution (must use CSS modules or Tailwind)
3. **HMR**: Slower hot-reload than Vite/React
4. **Talent Gap**: Fewer Dioxus developers than React
5. **Debugging**: WebAssembly debugging more challenging

### Mitigation Strategies

- **Component Library**: Build internal component library from day 1
- **Styling**: Use Tailwind CSS via CDN (Dioxus-friendly)
- **Training**: Invest in Rust/WASM training for frontend devs
- **Documentation**: Document Dioxus patterns and best practices

### Component Example

```rust
use dioxus::prelude::*;

#[component]
pub fn SchemaList(
    org_id: String,
) -> Element {
    let mut schemas = use_signal(Vec::new);
    let mut search = use_signal(String::default);

    use_future(move || {
        let org_id = org_id.clone();
        async move {
            let client = ApiClient::new();
            let result = client.list_schemas(&org_id).await;
            if Ok(s) = result {
                schemas.set(s);
            }
        }
    });

    rsx! {
        Layout {
            div { class: "p-6",
                h1 { class: "text-2xl font-bold", "Schemas" }
                input {
                    class: "border p-2 rounded",
                    placeholder: "Search...",
                    value: "{search}",
                    oninput: move |e| search.set(e.value())
                }
                div { class: "mt-4 grid grid-cols-3 gap-4",
                    schemas.iter().filter(|s| {
                        s.naam.contains(&*search.read())
                    }).map(|schema| {
                        rsx! {
                            SchemaCard {
                                key: "{schema._key}",
                                schema: schema.clone()
                            }
                        }
                    })
                }
            }
        }
    }
}
```

### API Client

```rust
use reqwest::Client;
use metadata_core::entities::Schema;

pub struct ApiClient {
    base_url: String,
    client: Client,
}

impl ApiClient {
    pub fn new() -> Self {
        Self {
            base_url: "/api/v2".to_string(),
            client: Client::new(),
        }
    }

    pub async fn list_schemas(&self, org_id: &str)
        -> Result<Vec<Schema>, ApiError>
    {
        let url = format!("{}/schemas?org_id={}", self.base_url, org_id);
        let response = self.client
            .get(&url)
            .send()
            .await?;

        Ok(response.json().await?)
    }
}
```
