## Context

The Metadata Registry Service must store the Dutch Government Metamodel GGHH V2 specification, which involves:
- **8 core entities** with complex relationships (Gebeurtenis, Gegevensproduct, ElementaireGegevensset, EnkelvoudigGegeven, WaardeMetTijd, Context, Grondslag, Zaak)
- **29 edge collections** for graph traversals
- **Time-based validity** (geldig_vanaf/geldig_tot) for all entities
- **Multi-tenancy** across 500+ government organizations
- **Graph queries** at depth 3-5 with <500ms response time

### Constraints

- **TC-2**: Must support graph queries natively (no post-processing joins)
- **TC-5**: Must comply with Sovereign Technology principle (open-source)
- **NFR-MREG-P-2**: Graph traversal (3 hops, 1000 nodes) <100ms
- **NFR-MREG-S-2**: Must scale to 100M entities over 5 years

### Current State

The existing IOU-Modern infrastructure has ArangoDB expertise available. No graph database has been deployed for production use yet.

---

## Decision

**Selected Option**: ArangoDB Community Edition

**ArangoDB will be used as the primary database for all entity storage, graph traversals, and time-based validity queries.**

---

## Alternatives Considered

### Option 1: ArangoDB (SELECTED)

**Pros**:
- Native multi-model (document, graph, key-value)
- AQL query language with graph traversal primitives
- Built-in time-based indexing for validity queries
- Open-source Community Edition (Apache 2.0)
- Horizontal scaling via cluster edition
- Strong consistency options
- JSON-first (aligns with GGHH V2 spec)

**Cons**:
- Smaller community compared to PostgreSQL
- Cluster edition requires commercial license
- Limited managed service options in NL
- Steeper learning curve for AQL

**Impact**: Optimal for GGHH V2 implementation with native graph support

---

### Option 2: PostgreSQL + Apache AGE

**Pros**:
- Larger talent pool in government
- Mature ecosystem, excellent tooling
- Apache AGE provides graph extensions
- Strong ACID guarantees
- Multiple managed service options

**Cons**:
- Graph operations via SQL extensions (less natural)
- Time-based validity requires custom implementation
- JSONB performance lower than native document store
- AGE graph traversals slower than native graph DB

**Impact**: Familiar technology but suboptimal for graph workloads

---

### Option 3: Neo4j Community

**Pros**:
- Purpose-built graph database
- Cypher query language (industry standard)
- Excellent graph traversal performance
- Large graph community

**Cons**:
- Community edition lacks horizontal scaling
- Commercial license required for clustering
- Document storage less natural than ArangoDB
- No native multi-model support

**Impact**: Excellent for graphs but requires separate document store

---

### Option 4: JanusGraph (with Cassandra/Scylla)

**Pros**:
- Distributed graph database
- Apache TinkerPop integration
- Scales horizontally
- Open-source (Apache 2.0)

**Cons**:
- Complex operational requirements
- Gremlin query language (less intuitive than Cypher/AQL)
- Requires separate storage backend
- Less mature than ArangoDB/Neo4j

**Impact**: High complexity for operational overhead

## Consequences

### Positive Consequences

1. **Native Graph Support**: AQL `FOR v, e, p IN 1..5 OUTBOUND ...` for deep traversals
2. **Multi-Model**: Single database for documents, graphs, and key-value lookups
3. **Time-Based Validity**: Persistent indexes on `geldig_vanaf`/`geldig_tot` for temporal queries
4. **JSON-First**: Direct mapping to GGHH V2 JSON schema
5. **Open-Source**: Community Edition aligns with Sovereign Technology

### Negative Consequences

1. **Cluster License**: Enterprise Edition required for horizontal scaling
2. **Talent Gap**: Fewer ArangoDB developers vs PostgreSQL
3. **Managed Services**: Limited ArangoDB Cloud options in Netherlands
4. **AQL Learning**: Query language less familiar than SQL

### Mitigation Strategies

- **Scaling Strategy**: Use database sharding at application layer before requiring Enterprise
- **Training**: Invest in ArangoDB certification for team
- **Documentation**: Create AQL query patterns and best practices guide
- **Hybrid Approach**: Consider PostgreSQL for operational data, ArangoDB for metadata

### Database Schema

```javascript
  "_key": "evt-001",

  "_to": "gegevensproduct/gdp-001",

  "geldig_tot": "9999-12-31T23:59:59Z"

```

### AQL Query Patterns

```aql
// Time-based validity lookup
FOR doc IN gebeurtenis
  FILTER doc.organisatie_id == @org_id
  FILTER doc.geldig_vanaf <= @now
  FILTER doc.geldig_tot >= @now
  RETURN doc

// Graph traversal with time validity
FOR v, e, p IN 1..3 OUTBOUND @start_id
  GRAPH 'gghh_v2'
  FILTER e geldig_vanaf <= @now
  FILTER e geldig_tot >= @now
  FILTER v geldig_vanaf <= @now
  FILTER v geldig_tot >= @now
  RETURN {vertex: v, edge: e, path: p}
```

### Index Strategy

```javascript
// Persistent indexes for time-based queries
db.gebeurtenis.ensureIndex({
  type: "persistent",
  fields: ["geldig_vanaf", "geldig_tot"],
  unique: false
});

// Organization index for multi-tenancy
db.gebeurtenis.ensureIndex({
  type: "persistent",
  fields: ["organisatie_id"],
  unique: false
});

// TTL index for automatic cleanup
db.gebeurtenis.ensureIndex({
  type: "ttl",
  fields: ["geldig_tot"],
  expiryDays: 365
});
```
