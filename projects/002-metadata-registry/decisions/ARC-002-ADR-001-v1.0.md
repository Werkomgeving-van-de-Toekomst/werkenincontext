## Context

The Metadata Registry Service requires a high-performance, type-safe implementation of the Dutch Government Metamodel GGHH V2 specification. The service must handle complex graph traversals, maintain strict audit trails, and serve concurrent API requests with sub-100ms response times. The Dutch government's Sovereign Technology principle requires the use of open-source technologies with strong security guarantees.

### Constraints

- **TC-1**: Must support high-concurrency (100+ concurrent API clients)
- **TC-2**: Must provide memory safety without garbage collection pauses
- **TC-3**: Must compile to WebAssembly for admin UI
- **TC-4**: Must have strong async/await support for I/O operations
- **BC-1**: Must comply with Sovereign Technology principle (open-source only)

### Current State

The existing IOU-Modern codebase uses Rust, and the initial Metadata Registry implementation was prototyped in Rust. No other languages have been evaluated for this service.

---

## Decision

**Selected Option**: Rust 2021 Edition

**Rust will be used as the primary language for all backend services (REST API, GraphQL API, GitOps sync, validation engine) and the WebAssembly-based admin UI.**

---

## Alternatives Considered

### Option 1: Rust (SELECTED)

**Pros**:
- Memory safety without garbage collection
- Zero-cost abstractions, performance comparable to C/C++
- Strong type system prevents entire classes of bugs
- Excellent async/await support (tokio)
- Compiles to WebAssembly for admin UI
- Growing ecosystem for government technology adoption
- Open-source, permissive licensing (MIT/Apache 2.0)

**Cons**:
- Steeper learning curve for developers
- Longer compile times than interpreted languages
- Smaller ecosystem compared to Java/JavaScript
- Hireability concerns in government sector

**Impact**: High performance, type-safe implementation aligned with Sovereign Technology principle

---

### Option 2: Java / Spring Boot

**Pros**:
- Large talent pool in government sector
- Mature ecosystem (Spring, Hibernate)
- Strong enterprise support
- Excellent observability tooling

**Cons**:
- JVM memory overhead and GC pauses
- More verbose than Rust
- Cannot compile to WebAssembly
- Oracle licensing concerns for sovereign deployment
- Higher resource consumption

**Impact**: Easier hiring but higher infrastructure costs and performance concerns

---

### Option 3: Go / golang

**Pros**:
- Simple syntax, fast compilation
- Excellent concurrency primitives (goroutines)
- Strong standard library
- Good performance

**Cons**:
- No WebAssembly support for admin UI
- Weaker type system than Rust
- No compile-time memory safety guarantees
- Error handling philosophy differs from government practices

**Impact**: Good performance but lacks WebAssembly and stronger type safety

---

### Option 4: TypeScript / Node.js

**Pros**:
- Largest talent pool
- Full-stack (backend + frontend)
- Fast development iteration
- Rich ecosystem

**Cons**:
- V8 memory overhead and GC pauses
- No compile-time type safety (types erased at runtime)
- Performance limitations for graph traversals
- Higher runtime security surface

**Impact**: Fast development but performance and security concerns for government data

## Consequences

### Positive Consequences

1. **Performance**: Sub-100ms API response times achievable with zero-copy deserialization
2. **Memory Safety**: Eliminates entire classes of security vulnerabilities (use-after-free, buffer overflows)
3. **WebAssembly**: Admin UI runs in browser with native performance
4. **Sovereign Technology**: Fully open-source stack aligns with government principles
5. **Maintainability**: Strong type system prevents bugs at compile time
6. **Deployment**: Single binary deployment reduces attack surface

### Negative Consequences

1. **Learning Curve**: 3-6 months for Java/TypeScript developers to reach productivity
2. **Compile Times**: 30-60 second incremental compiles during development
3. **Hiring**: Smaller pool of Rust developers in government sector
4. **Ecosystem**: Fewer government-specific libraries compared to Java

### Mitigation Strategies

- **Training**: Invest in Rust training programs for existing developers
- **Hiring**: Partner with specialized Rust recruitment agencies
- **Documentation**: Extensive inline documentation and architecture diagrams
- **Tooling**: Configure cargo build caching to reduce compile times
